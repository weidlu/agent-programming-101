# Agent 设计模式

## 为什么需要设计模式？

单一 Agent 在复杂任务上有天花板：
- 上下文窗口限制（处理不了超长文档）
- 无法并行（只能顺序思考）
- 专业性不足（一个模型擅长所有领域？）

设计模式提供了组织多 Agent 的标准方式。

## 模式一：ReAct（推理-行动循环）

最基础的 Agent 模式，每一步循环：Reason（推理）→ Act（行动）→ Observe（观察结果）

```
用户: "帮我找北京明天的天气，然后推荐穿什么"

Step 1: Reason: 需要查天气
        Act: call_tool(get_weather, city="北京", date="明天")
        Observe: 晴，最高12°C

Step 2: Reason: 已有天气信息，可以推荐穿着
        Act: [输出最终回答]
        Observe: done
```

Phase 1 的 while loop 实现的就是 ReAct 模式。

## 模式二：Plan-and-Execute

先生成完整计划，再逐步执行。适合复杂多步骤任务。

```
Planner: ["查北京天气", "查上海天气", "对比两城市", "生成报告"]
Executor: 顺序执行每个步骤
```

**优点**：可以并行执行无依赖的子任务  
**缺点**：计划可能不准，中途需要调整

## 模式三：Supervisor-Worker（主管-工人）

一个 Supervisor 负责任务分发和质量把控，多个 Worker 各司其职。

```
用户需求 → Supervisor
                ↓ 分配
    ┌──────────┴──────────┐
Researcher              Writer
(查资料)                (写文章)
    └──────────┬──────────┘
               ↓ 汇报
           Supervisor（质量把关 → 如不满意返工）
               ↓
           最终输出
```

Phase 4 将实现这个模式（研报生成团队）。

## 模式四：RAG Agent（检索增强）

这是 **Phase 3** 的核心：把"查知识库"作为一个 Tool，让 LLM 自主决定何时检索。

```
用户问题 → LLM（判断是否需要检索？）
                ↓ 是
      call_tool(search_notes, query="...")
                ↓
      检索结果 → LLM（根据结果回答）
                ↓
            最终回答
```

**和普通 RAG 的区别**：
- 普通 RAG：每次都检索，无论是否需要
- RAG Agent：LLM 自主决定检索时机，更灵活

## 核心设计原则

### 最小授权
工具只能访问必要的资源。Agent 不应该有权限操作它不需要碰的东西。

### 幂等性
特别是在有 Human-in-the-loop 的场景，节点可能被重复执行。有副作用的操作应该有幂等守卫。

### 可观测性
Agent 的中间过程必须可追踪。LangSmith、日志文件、或者简单的 print 都可以，
关键是能"回放"每一步发生了什么。

### 渐进式复杂度
先把单 Agent + 工具跑通，再加多 Agent。
不要一开始就设计 5 个 Agent 互相通信，那会调试到崩溃。
